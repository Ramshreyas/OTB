Resolution Pipeline

    OTB Weather Market Resolver — given a prediction market case (Polymarket URL, ancillary data with Wunderground station rules), retrieves public weather evidence, reconciles it against market rules, and returns p1 (No), p2 (Yes), p3 (50/50), p4 (Too Early), or unclear.

    Two entry points, one pipeline: resolve.py processes a markets.json manifest (replay or live). resolve_otb.py polls the OTB Oracle API for live proposed Weather markets and runs the same resolver pipeline — paper-proposing recommendations without affecting settlement.

    Design principles: Source authority is explicit (station URL, never city name). Units are verified, never assumed. Conservatism over confidence — a wrong confident answer is worse than unclear. Every decision is traceable via Langfuse spans.

    Schema validation — what gets checked

    Each market case must have: case_id, polymarket_url, question_data.outcomes (p1/p2/p3/p4), ancillary_data (non-empty, with station URL and finality rules), and question_data.end_date_iso (parseable date). Schema lives in data/schema/market_input.schema.json.

    Compose Retrieval Spec

    src/retrieval/spec.py, guardrails.py, llm_extractor.py

    What it does

    Takes free-text ancillary_data (written as instructions for a human with a browser) and produces a structured RetrievalSpec that tells the retrieval layer exactly what to fetch.

    • LLM extracts: station URL, station code, measurement type, aggregation (min/max/sum), unit, precision, timezone, finality window
    • Regex fallback if LLM unavailable or returns non-conforming output
    • Station registry enriches with timezone, known quirks, guardrails. Unknown stations log a warning but don't gate — they proceed with LLM-extracted metadata only (no timezone verification, no guardrails). In production, an unrecognized station in a live OTB market would trigger operator escalation.
    • Cross-validates against question_data — catches title/ancillary contradictions
    Extraction layers — LLM → schema validation → regex → station registry

    Layer 1: LLM extraction — single call with structured output schema. The LLM reads ancillary_data as intended: human-readable instructions with embedded URLs, UI navigation tips, and prose finality rules. It understands negation ("do not use Narita"), context ("use daily high, not intraday"), and intent.

    Layer 2: Schema validation — every LLM response is validated. Fields must match expected types, URLs must be syntactically valid, measurements must be from known enums.

    Layer 3: Regex fallback — deterministic safety net for LLM unavailability. Matches Wunderground URL patterns, ISO dates, known unit phrases. Flagged in trace — regex-derived specs are capped at 0.70 confidence and always invoke the reviewer. Regex cannot understand negation or prose context, making it vulnerable to crafted ancillary data. The reviewer + confidence cap + operator visibility mitigate this: a regex extraction on a high-stakes market draws attention.

    Layer 4: Station registry — resolves bare ICAO codes to full URLs, maps stations to IANA timezones, attaches guardrails (retry guidance, partial-data windows, source lag tolerance).

    Retrieval

    src/retrieval/dispatch.py, wunderground_api.py, wunderground_playwright.py, replay.py

    What it does

    Fetches raw weather observations using the RetrievalSpec. Dispatches by source type — Wunderground for daily temperature markets, NOAA for monthly precipitation markets. Two modes:

    • Live — Wunderground: direct API → Playwright headless browser fallback → exhaustion → unclear. NOAA: monthly summary API → exhaustion → unclear (no browser fallback — NOAA doesn't require JS rendering).
    • Replay — loads pre-captured fixture from data/fixtures/. No network calls. Deterministic.

    Live mode auto-saves the response as a fixture — the system bootstraps its own determinism for future replay. Every retrieval records exact URL, HTTP status, latency, path taken, and any errors in source_trace.

    Input → Output

    Input
    RetrievalSpec (from Stage 2)
    Output
    RawObservationBatch — observations, extracted value, finality flag, source trace
    Retrieval flow — 16-step process driven by RetrievalSpec fields

    Each field in the RetrievalSpec drives a specific action: source_type dispatches to the right strategy (wunderground_station → API, noaa_monthly → NOAA). station_url extracts ICAO code + country. target_window + timezone build date range in station-local time. unit sets API unit parameter. guardrails.retry configures retry policy. measurement + aggregation look up the API field mapping (e.g., temperature+min → field=temp, fn=min). A second API call for finality_after checks if next-day data exists. The output is an immutable RawObservationBatch with observations, extracted value, finality status, and full source trace (URL, HTTP status, latency, path, retries, guardrail flags, errors).

    Why city name is not station — the Denver/Buckley problem

    The market title says "Denver" but the resolution source is Buckley Space Force Base (KBKF) in Aurora, CO. "Seoul" resolves to Incheon Intl (RKSI). The RetrievalSpec always uses the exact Wunderground URL from ancillary data — never derives a station from a city name. If ancillary_data contains only a station name without a URL, the spec stage constructs the URL from the station registry or flags the case as unclear.

    Normalization

    src/normalization/convert.py, round.py, quality.py, anomaly.py

    What it does

    Transforms raw observations into market-comparable values. Every operation is pure math or rule-based — no LLM. Deterministic by design.

    • Unit conversion: °C ↔ °F, in ↔ mm. Verifies response unit label, never assumes.
    • Precision rounding: whole degrees (temp) or 2 decimals (precip)
    • Quality checks: completeness (actual/expected observations), temporal gaps, guardrail flag escalation
    • Anomaly detection: physical limits (e.g., temperature outside [-80, 60]°C → flag)

    Gating decisions

    CheckIf it fails
    At least one observation present?unclear — no observations in window
    Unit conversion possible?unclear — unit provenance conflict
    No critical anomaly? (value outside physical limits)unclear — anomalous data

    Quality flags (partial data, gaps) are soft — they reduce confidence but don't gate. Only hard failures gate to unclear.

    Anomaly thresholds — physical limits, not statistical outliers

    Fixed physical bounds, not z-scores or neighbor comparisons. An unusual-but-real temperature (40°C in Tokyo) is not flagged. An impossible temperature (90°C) is. Ranges: temperature [-80, 60]°C, wind gust [0, 400] kph, humidity [0, 100]%, precipitation [0, 3000] mm/month, pressure [850, 1090] hPa.

    Reconciliation

    src/reconciliation/rule_parser.py, finality.py, comparator.py

    What it does

    Three sequential gates — any can short-circuit to p4 or unclear before reaching the threshold comparison.

    Gate 1: Finality

    Finality flagAction
    confirmedPass. Proceed to gate 2.
    not_yetp4: Too Early — next-day datapoint doesn't exist yet.
    unknownp4: Too Early — conservative: finality fetch failed.

    Gate 2: Quality

    FlagSeverityAction
    partial_data / observation_gapSoftPass, reduce confidence
    unit_provenance_conflictSoftPass, reduce confidence
    Completeness < 0.5Hardunclear
    Physical anomaly detectedHardunclear

    Soft flags accumulate confidence penalties (~0.1 each). Hard flags gate immediately.

    Gate 3: Parse Rules & Compare

    Extract operator and threshold from question title (regex, not LLM — patterns are constrained). Apply bracket/tie rules from res_data in ancillary data.

    Question patternOperatorThreshold
    "be 20°C on June 1?"exact (==)20
    "be 29°C or higher?"gte (≥)29
    "be 22°C or below?"lte (≤)22
    "be between 68-69°F?"range (∈)[68, 69]

    Comparison is pure math on known numbers. Clear result → Yes/No. Ambiguous boundary → p3: 50/50. Unparseable rules → unclear.

    Decision

    src/decision/resolver.py, confidence.py, reviewer.py

    What it does

    • Deterministic mapping: Yes → p2, No → p1, p3_tie → p3, p4_too_early → p4, unclear → unclear
    • Confidence scoring: starts at 1.0, applies penalties from quality flags. Source path matters: API → no penalty, Playwright → -0.1, Replay → no penalty.
    • LLM reviewer (conditional): fires only when confidence < 0.85. The reviewer sees the full evidence chain but can only escalate to unclear — it cannot substitute its own p1/p2 answer. If the reviewer call fails, it defaults to agreeing with the deterministic result.
    Confidence scoring breakdown

    Starts at 1.0. Penalties: partial_data = -0.1, observation_gap = -0.1, unit_provenance_conflict = -0.15, near_day_boundary = -0.05. Source path: API = 0, Playwright = -0.1. Minimum confidence = 0.1. For unclear/p4 verdicts, confidence is always 1.0 (we're certain it's unclear/too-early).

    LLM reviewer — safety valve, not a decision-maker

    The reviewer prompt is managed in Langfuse Prompt Registry as weather-reviewer (label: production). It receives the full evidence chain: station, value, unit, completeness, quality flags, deterministic recommendation, and reasoning. It must answer with {"agree": true} or {"agree": false}. If it disagrees, the recommendation becomes unclear. It cannot propose a different p1/p2. If the reviewer call fails (network, model unavailable), it defaults to agreeing — conservatism with availability.

      Langfuse Tracing

      src/observability/tracing.py, steps.py, llm.py

      How tracing works

      • @step decorator — wraps every pipeline stage. Creates a Langfuse span with rich input (what the stage received) and output (what it produced). ~30 lines, applied automatically to any stage defined in pipeline.yaml.
      • langfuse.openai.OpenAI — drop-in replacement for the OpenAI client. Every LLM call is auto-captured as a GENERATION nested under the current span, with model, tokens, latency, and I/O. No manual instrumentation.
      • Prompt linking — generations link back to the exact prompt version in Langfuse. From a trace, click the prompt link → see the prompt content used for that specific LLM call. Full audit trail.

      Prompt Registry

      PromptUsed byDescription
      weather-spec-extractionStage 2 (compose_spec)Extracts structured fields from free-text ancillary_data. Chat prompt with template variables: {{title}}, {{ancillary_data}}.
      weather-reviewerStage 6 (decide, conditional)Reviews the full evidence chain when confidence < 0.85. Can only escalate to unclear. Template: {{recommendation}}, {{confidence}}, {{value}}, {{quality_flags}}, etc.

      Prompts are versioned with labels (production, staging, latest). The resolver fetches whatever version has the production label. To roll back a bad prompt: change the label. Zero code deploy. See scripts/seed_langfuse_prompts.py.

      LiteLLM Proxy

      litellm_config.yaml, src/observability/llm.py

      What it does

      • Provider-agnostic routing — the resolver calls model="deepseek-chat" but LiteLLM maps it to deepseek/deepseek-chat per config
      • Switch providers by changing one line in litellm_config.yaml — resolver code doesn't change
      • Rate limiting (30 RPM cap), cost tracking, per-request logging
      • Model fallback: if DeepSeek is down, route to Gemini or Claude (configured in YAML)

      Config — litellm_config.yaml

      model_list: - model_name: deepseek-chat litellm_params: model: deepseek/deepseek-chat api_key: ${DEEPSEEK_API_KEY} rpm: 30 litellm_settings: master_key: sk-litellm-otb-master-key

      Next Steps — Production Scaling

      See docs/next-steps.md for full discussion

      Operational observability

      • Aggregated dashboards: source freshness, failure rates, confidence drift, unclear rate trends
      • Per-station monitoring: if KBKF degrades while RJTT is fine, flag the station, not the pipeline
      • Settlement reconciliation: compare paper-propose against actual Polymarket/DVM outcomes

      Human circuit breaker

      • Escalation triggers: confidence < 0.70 after reviewer, reviewer disagrees, new source type, bulletin board update, retrieval exhausted
      • Operator actions: approve, override, mark unclear, re-retrieve. Every action audited.
      • Circuit breaker: if >N cases escalate in M minutes, stop auto-resolution

      Scaling

      • Model fallback: DeepSeek → Gemini → Claude → regex-only (no LLM)
      • Priority queueing: settlement deadline × log(market size). Critical tier for >$100K markets <4h to deadline.
      • Persistent state: checkpoint after each stage. Resume from checkpoint on container restart.
      • Canary queries: periodic test against known-good stations. If canary fails, pause live resolution.

      Benchmark expansion

      • Capture settled Polymarket markets to grow benchmark beyond 5 cases
      • Add edge cases: p4 (too early), unclear (corrupted data), DST transitions, bulletin board updates
      • Continuous evaluation: weekly run against full benchmark. Flag regressions immediately.

      Preventing repeat mistakes

      • Every production incident → new benchmark case that reproduces it → regression test
      • Every new station quirk → guardrail update in station registry
      • Every new failure mode → monitoring rule for early detection
