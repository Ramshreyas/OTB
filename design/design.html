Resolution Pipeline

    Input & Validation

    src/validation/loader.py, schema.py, models.py

    What it does

    • Loads markets.json from the --input path
    • Validates every case object against the input schema
    • Fails fast if any case is malformed — no partial processing
    • Each validated case is loaded into an immutable data structure

    Input → Output

    Input
    markets.json (manifest with case_id, question_data, ancillary_data)
    Output
    List[MarketCase] — immutable, validated
    Schema validation — what gets checked

    Each market case must have:

    • case_id — unique string identifier
    • polymarket_url — valid URL
    • question_data.outcomes — p1/p2/p3/p4 labels present
    • ancillary_data — non-empty string with station URL, unit, precision, finality rule
    • question_data.end_date_iso — parseable date

    Schema lives in data/schema/market_input.schema.json. Missing required fields or malformed dates cause immediate rejection with a clear error message.

    Why immutability matters

    Market cases are frozen after load. Every subsequent pipeline stage builds a parallel resolution data structure rather than mutating the original. This ensures:

    • The source of truth is preserved for trace/debug
    • Replay runs produce identical intermediate states
    • Concurrent processing (future) is safe by default

    Compose Retrieval Spec

    src/retrieval/spec.py, guardrails.py, source_detector.py

    What it does

    Takes the validated market case and produces a complete, structured RetrievalSpec that tells the retrieval layer exactly what to fetch, where, and with what guardrails.

    • Feeds the full ancillary_data string and question_data.title to an LLM with a structured output schema to extract the station URL, target window, measurement type, unit, precision, finality threshold, and source type
    • Validates the LLM's output against the RetrievalSpec schema; falls back to regex extraction if the LLM is unavailable or returns non-conforming output
    • Runs a station registry lookup to enrich the result with timezone, full URL construction from bare ICAO codes, and known station metadata
    • Attaches station-specific guardrails — known quirks, timing expectations, and retry guidance — that the retrieval layer uses to harden its strategy

    How parsing works

    ancillary_data is free-text written as instructions for a human browsing a weather website. It can embed URLs mid-paragraph, include UI navigation tips, state finality rules in prose, and vary in structure across markets. Extraction is LLM-first with deterministic fallback:

    LayerMethodWhat it handles
    1. LLM extractionSingle LLM call with a structured output schemaThe primary path. The LLM receives the full ancillary_data string and question_data.title, and is prompted to return the structured fields (source type, station URL, target window, measurement type, unit, precision, finality after). The prompt includes the exact schema, field descriptions, and examples. Because ancillary_data is already human-readable instructional text, an LLM can read it as intended — it understands "the highest temperature recorded … available here: <url>" without needing a URL regex, and can infer the window from phrases like "on 1 Jun '26" (single day) or "for May 2026" (full month).
    2. Schema validationValidation of LLM output against RetrievalSpec schemaEvery LLM response is validated. Fields must match expected types, URLs must be syntactically valid, dates must be parseable, measurement types must be from a known enum. If validation fails, the output is discarded and extraction falls back to layer 3.
    3. Regex fallbackPre-compiled regex patterns per fieldDeterministic safety net. Used when the LLM is unavailable, returns non-conforming output, or as a cost-saving fast path for simple cases. Matches Wunderground/NOAA URL patterns, ISO dates, known unit phrases, precision phrases.
    4. Station registryLookup table of known ICAO codesAugments extracted results regardless of which layer produced them. Resolves bare ICAO codes to full URLs and maps stations to IANA timezones. Also used as a cross-reference: if the LLM extracts "RJTT" as the station code, the registry confirms it and provides the timezone.

    Layer 1 is the default. Layers 3–4 are always available as fallback, but they are not invoked when the LLM produces a valid, schema-conforming result. The station registry (layer 4) always runs after extraction to enrich the result, regardless of which layer produced it.

    Cross-validation with question_data

    After extraction, the retrieved fields are cross-checked against question_data to catch disagreements between what ancillary_data describes and what the market metadata asserts. A mismatch does not automatically block resolution, but it reduces confidence and may gate to unclear if the conflict is fundamental.

    CheckSources comparedIf they disagree
    Date/window agreementExtracted target window vs. end_date_isop4-adjacent — if end_date_iso falls within the extracted window, they agree. If they reference different calendar days or non-overlapping ranges, the case gates to unclear — the market's own metadata contradicts its resolution instructions. If the window is a month but end_date_iso is the first day of that month, the window is authoritative (NOAA monthly summaries resolve against the full month, not a single day).
    Unit consistencyExtracted unit vs. unit implied by question title (e.g., title says 20°C but ancillary_data describes Fahrenheit)Flagged — title and ancillary_data describe different units. The ancillary_data unit takes precedence (it's the resolution source), but the mismatch is recorded in the trace and confidence is reduced. Post-retrieval, the actual observation unit is verified.
    Measurement consistencyExtracted measurement type vs. title keywordsFlagged — title says "lowest" but ancillary_data extraction returned daily_high. The LLM extraction is re-prompted with both texts side-by-side. If the conflict persists, gates to unclear.
    Station ⇔ city awarenessStation code in URL vs. city name in titleNoted — does not gate, but recorded in trace for operator visibility. "Denver" resolving to KBKF in Aurora is correct per ancillary_data, but an operator reviewing the trace should see this flagged so they understand why the station differs from the city in the title.

    Guardrails — operational hints for the retrieval layer

    After extraction, the RetrievalSpec is augmented with station-specific notes drawn from a curated registry of known behaviors. The retrieval layer reads these to adjust its strategy:

    Guardrail typeExample
    Unit-label verification"RJTT returns °C regardless of query param — verify response unit label, not request param"
    Partial-data windows"KBKF may serve intraday values before 06:00 local — if retrieval is near day boundary, flag incomplete"
    Source lag tolerance"NOAA monthly summaries lag 3–5 days after month-end — extend effective finality date"
    Expected response shape"Daily high/low appears in a 24-row observations table — fewer rows = partial data; flag in quality check"
    Retry guidance"This station is known to 503 under load — retry up to 3× with 5s backoff before falling back to Playwright"
    City name is not station — the Denver/Buckley problem

    The market title often says "Denver" but the resolution source is Buckley Space Force Base (KBKF) in Aurora, CO. "Seoul" resolves to Incheon Intl (RKSI), not a Seoul city station. The RetrievalSpec always uses the exact Wunderground URL from ancillary_data, never derives a station from the market title.

    If ancillary_data contains only a station name without a URL, the spec stage must construct the URL from known station registries (ICAO → Wunderground path mapping) or flag the case as unclear. It must not guess a station from a city name.

    LLM extraction: the primary path

    ancillary_data is written as instructions for a human with a browser. It describes where to go, what to look for, and how to interpret it — exactly the kind of text an LLM reads well.

    The LLM receives the full ancillary_data string plus question_data.title and is prompted to return a structured JSON object with all RetrievalSpec fields. The prompt includes the exact output schema, field-by-field descriptions, and 2–3 annotated examples drawn from real markets. The LLM is asked to return null for any field it cannot determine.

    The response is validated against the RetrievalSpec schema. Fields the LLM left as null trigger the regex fallback for those specific fields only. A completely non-conforming response (wrong types, hallucinated URLs) is discarded entirely and the regex fallback handles the whole extraction. The LLM gets one call per market — no iterative prompting.

    Step-by-step: how the RetrievalSpec drives each fetch

    Below is the procedural walkthrough for a wunderground_station market — the most common and complex path. Each step shows which spec field is consumed and what action it drives.

    StepSpec field consumedAction
    1. Choose strategysource_typewunderground_station → dispatch to wunderground_api.py. noaa_monthly → dispatch to noaa.py. Unknown → unclear.
    2. Parse station identitystation_urlExtract ICAO code and country from the URL path: …/jp/tokyo/RJTT → ICAO=RJTT, CC=JP. If the URL can't be parsed, fall back to station registry lookup.
    3. Build date rangetarget_window + timezoneConvert window start/end from station-local to YYYYMMDD format for the API. A single day: startDate=endDate=20260601. A month: startDate=20260501&endDate=20260531. The timezone ensures the date boundaries are correct — June 1 JST starts at May 31 15:00 UTC.
    4. Set unit parameterunitMap to API units param: C → m (metric), F → e (imperial). The API URL becomes: GET …/observations/historical.json?apiKey=…&units=m&startDate=20260601&endDate=20260601.
    5. Apply retry guardrailsguardrails.retryRead the retry guidance: "3× with 5s backoff for 503". Configure the HTTP session accordingly. If all retries are exhausted → fall back to Playwright.
    6. Execute primary fetch(all above combined)Send the HTTP request. Record: URL, status code, latency, response size, retry count. On success → step 7. On failure after retries → Playwright fallback (or unclear if no fallback exists).
    7. Validate response shapeguardrails.response_shapeCount observations in the response. A single-day window should have ~24. If the count is significantly lower (e.g., 7), flag as partial and record the guardrail trigger. A partially-populated day at the current date is expected; a fully missing historical date is suspicious.
    8. Verify unit labelguardrails.unit_verificationCheck the response metadata for the actual unit returned. Some stations (RJTT) return °C regardless of the units param. If response unit ≠ requested unit, record the mismatch for normalization to resolve.
    9. Filter by timezonetimezone + target_windowConvert every observation's valid_time_gmt (UTC epoch) to station-local time using the IANA timezone. Discard observations that fall outside the target window. A June 1 JST window will include observations from 2026-05-31T15:00Z through 2026-06-01T14:59Z.
    10. Look up field mappingmeasurement + aggregationQuery the field mapping table in dispatch.py: temperature + min → extract temp from each observation, apply min(). wind_gust + max → extract gust, apply max(). precipitation + sum → extract precip_total, apply sum().
    11. Aggregate(field mapping result)Extract the target field from each in-window observation. Apply the aggregation function. Record the result: {"value": 18, "unit": "C", "field": "temp"}.
    12. Build finality fetchfinality_afterConstruct a second API call for the next-day window: startDate=endDate=20260602. Apply the same retry guardrails. If the fetch returns ≥1 observation with a timestamp ≥ finality_after, finality is confirmed. Otherwise not_yet. If the finality fetch itself fails after all retries and Playwright fallback, finality is unknown.
    13. Apply source-lag guardrailguardrails.source_lagFor sources with known lag (NOAA: +5 days), if today's date is within the lag window, extend the effective finality threshold. The finality check still runs, but a not_yet result is treated as expected rather than a hard gate.
    14. Check partial-data windowguardrails.partial_dataIf the primary fetch occurred within the partial-data window (e.g., within 6 hours of the window end per guardrail config), and the observation count is below the expected threshold, flag the batch as potentially incomplete. Normalization will apply extra scrutiny.
    15. Emit RawObservationBatch(all)Package everything into an immutable RawObservationBatch: the filtered in-window observations, the aggregated value, the finality flag, and the full source trace with all guardrail triggers.
    16. Save fixture (live mode)(all above)Write the raw API response payloads (both primary and finality fetches), the Playwright-scraped data (if fallback was used), and the source trace to data/fixtures/<case_id>.json. This is the deterministic replay anchor — future replay runs load this file instead of making network calls, and produce identical results. The fixture contains: the raw HTTP response body (or scraped HTML), the extracted observations, the source trace, and the RetrievalSpec that produced it — so an operator can understand why the resolver fetched what it did.

    Playwright fallback path (wunderground_station only)

    When the API fails or returns unusable data, the Playwright strategy takes over. It uses the station_url directly rather than constructing an API call:

    • Launches a headless Chromium browser
    • Navigates to the full station URL from the RetrievalSpec
    • Clicks the gear icon to toggle the unit setting per unit (C ↔ F)
    • Waits for the observations table to render
    • Scrapes all rows — each row is an hourly observation
    • Normalizes the scraped fields to match the same shape as the API response, so downstream stages don't need to know which path was taken
    • Records in source trace: path=playwright, page load time, any rendering errors

    Playwright has its own timeout and retry logic (separate from the HTTP retry guardrails, since browser failures are different from API failures). If Playwright also fails, the fetch is exhausted → unclear.

    Why retrieval failures produce unclear (not guesses)

    If all strategies for a source type are exhausted, the resolver returns unclear rather than guessing. A wrong confident p1/p2 based on missing data is strictly worse than admitting the data wasn't retrievable. The system does not hallucinate weather observations.

    Normalization

    src/normalization/convert.py, round.py, verify.py, quality.py, anomaly.py

    What it does

    Takes the RawObservationBatch from retrieval and the RetrievalSpec from stage 2, and transforms the extracted value into a normalized form ready for comparison against market rules. Every operation is deterministic — unit conversion, precision rounding, window verification, and quality checks are all pure math or rule-based logic. No LLM involvement.

    Inputs

    From retrieval
    RawObservationBatch — observations array, extracted_value, finality flag, source_trace with guardrail flags
    From stage 2
    RetrievalSpec — target window, unit, precision, timezone, measurement, aggregation, guardrails

    Step-by-step transformation

    StepModuleWhat it doesDeterministic?
    1. Unit conversionconvert.pyIf the extracted value's unit differs from the spec's expected unit, convert. °F → °C: (F - 32) × 5/9. °C → °F: C × 9/5 + 32. in → mm: in × 25.4. mph → kph: mph × 1.609. All conversions are pure math with known constants. The converted value is tagged with the target unit.✅
    2. Precision roundinground.pyRound the converted value to the spec's precision. precision=1 (whole degrees) → round(v). precision=2 → round(v, 2). Note: this rounds the aggregated value, not individual observations — the station's own reported daily high/low is already whole-degree and was extracted as-is. Rounding is a safety net, not a transformation of station data.✅
    3. Window-boundary verificationverify.pyConfirm that every observation used in the aggregation falls within the target window per the spec's timezone. This is defense-in-depth — retrieval already filtered by timezone (step 9), but normalization re-verifies. If any in-window observation has a local timestamp outside the window, the mismatch is flagged in the trace.✅
    4. Guardrail flag escalationquality.pyRead the guardrail flags recorded in the source trace — partial data, unit mismatch, response shape violations — and escalate them into quality flags on the normalized observation. A unit mismatch flagged by retrieval becomes a unit_provenance_conflict quality flag here. A partial-data flag becomes incomplete_observations.✅
    5. Observation completenessquality.pyCount in-window observations vs. expected: a full single day should have ~24 hourly observations. Fewer → flag. Check for temporal gaps: if hour 03 and hour 05 are present but hour 04 is missing, flag as observation_gap. Check that the aggregation makes sense: a max over 3 observations is less reliable than over 24.✅
    6. Anomaly detectionanomaly.pyRun sanity range checks on the extracted value. A reported 90°C at Tokyo Haneda is a sensor error, not weather. Known physical limits per measurement: temperature outside [-80, 60]°C, wind gust > 300 kph, humidity outside [0, 100]%, precipitation > 2000 mm/month. Values outside these bounds are flagged as anomalous. This is not an LLM judgment call — it's a threshold check against known physical maxima.✅
    7. Emit NormalizedObservation__init__.pyPackage the normalized value, unit, precision, quality flags, completeness score, and anomaly flags into an immutable NormalizedObservation ready for reconciliation.✅

    Gating decisions

    These are hard gates. If any fail, the case cannot proceed to reconciliation:

    CheckIf it fails
    Unit conversion possible? (both source and target units known, conversion factor exists)unclear — unit provenance conflict
    Precision rounding unambiguous? (precision value is a positive integer, not 0 or negative)unclear — ambiguous precision
    At least one observation present within the target window?unclear — no observations in window
    No critical anomaly? (value outside physical possibility, not just unusual)unclear — anomalous data

    Quality flags (partial data, gaps, unit mismatches) are soft — they reduce confidence in the decision stage but do not gate to unclear by themselves. Only hard failures (no observations, impossible values, unconvertible units) gate.

    Why normalization is deterministic (no LLM)

    By the time data reaches normalization, all ambiguity has been resolved upstream. Stage 2 used an LLM to parse free-text ancillary data into a structured RetrievalSpec. Retrieval used that spec to fetch structured data with known field names and unit labels.

    Normalization's job is pure transformation on structured inputs: unit conversion (math), rounding (math), window verification (date arithmetic), completeness counting (arithmetic), and anomaly detection (threshold checks against known physical limits). There is no judgment call that an LLM would improve — it would only add latency, cost, and non-determinism to operations that are provably correct when done with math.

    Unit conversion safety — never assume, always verify

    The retrieval layer tags every extracted value with the unit it was returned in, not the unit it requested. If the API was asked for units=m (Celsius) but the response metadata says units=e (Fahrenheit) — because the station ignores the param — normalization sees the actual unit and converts accordingly.

    This means normalization does not trust that retrieval "got the right unit." It reads the unit label from the response and converts if needed. The guardrail flag unit_provenance_conflict is set when the response unit differs from the requested unit, and this travels downstream to reduce confidence in the decision stage.

    Anomaly thresholds — physical limits, not statistical outliers

    Anomaly detection uses fixed physical bounds, not statistical methods (no z-scores, no neighbor comparisons). Rationale: weather is inherently variable, and an unusual-but-real temperature (40°C in Tokyo) should not be flagged just because it's a 3-sigma outlier. An impossible temperature (90°C) should be flagged regardless of distribution.

    MeasurementValid rangeRationale
    Temperature[-80, 60] °CColdest recorded: -89.2°C (Vostok). Hottest: 56.7°C (Death Valley).
    Wind gust[0, 400] kphHighest recorded: 408 kph (Barrow Island cyclone, 1996).
    Humidity[0, 100] %Physical definition. Values >100% are sensor malfunctions.
    Precipitation[0, 3000] mm/monthHighest recorded monthly: ~9,300 mm (Cherrapunji, but that's annual). Monthly extreme ~2,500 mm.
    Pressure[850, 1090] hPaSea-level range. Highest: 1083.8 hPa (Agata, Siberia). Lowest: 870 hPa (Typhoon Tip).

    Reconciliation

    src/reconciliation/rule_parser.py, finality.py, quality_gate.py, comparator.py

    What it does

    Takes the NormalizedObservation from stage 4, the finality flag from retrieval, and the market's question text, and determines whether the observed value satisfies the market rules. Three sequential gates — any can short-circuit to p4 or unclear before reaching the threshold comparison.

    Inputs

    From normalization
    NormalizedObservation — value, unit, precision, quality_flags, anomaly_flags, completeness
    From retrieval
    Finality flag — confirmed, not_yet, or unknown
    From market case
    question_data.title + ancillary_data (res_data bracket rules)

    Gate 1: Finality

    The market rules state: "This market can not resolve until the first data point for the following date has been published." This is checked first — nothing else matters if the market isn't final.

    Finality flagAction
    confirmedPass. Proceed to gate 2.
    not_yetp4: Too Early — the next-day datapoint does not yet exist on the resolution source.
    unknownp4: Too Early (conservative) — the finality fetch failed, so we cannot confirm the market is final. Conservative default: assume it isn't.

    Note: the finality check data was already fetched by retrieval (the second API call). Reconciliation only reads the flag — it does not make network calls.

    Gate 2: Quality & Anomaly

    Before comparing values, check whether the normalized observation is trustworthy enough to compare. Quality flags from normalization are evaluated here.

    FlagSeverityAction
    No flags—Pass. Proceed to gate 3.
    partial_dataSoftPass, but reduce confidence. Fewer than expected observations exist for the window.
    observation_gapSoftPass, but reduce confidence. Temporal gaps in the observation series.
    unit_provenance_conflictSoftPass, but reduce confidence. The API returned a different unit than requested.
    near_day_boundarySoftPass, but reduce confidence. Retrieval occurred near the day boundary; data may be incomplete.
    value_out_of_physical_rangeHardunclear — the observed value is physically impossible (e.g., 90°C in Tokyo)
    sensor_error_suspectedHardunclear — the sensor appears to be malfunctioning
    Completeness < 0.5Hardunclear — fewer than half the expected observations; aggregation is unreliable

    Soft flags never gate on their own — they accumulate confidence penalties. A single soft flag drops confidence by ~0.1. Multiple soft flags compound. Hard flags or critically low completeness gate immediately.

    Gate 3: Parse Market Rules & Compare

    Extract the comparison operator and threshold from the question title, then apply them to the normalized value. Bracket and tie-breaking rules come from res_data in ancillary_data.

    3a. Operator & threshold extraction

    The question title is fed to a regex-based parser (not LLM — the patterns are constrained and well-known). The RetrievalSpec's measurement, aggregation, and unit fields provide cross-validation: the threshold's unit must match the spec's unit.

    Question patternOperatorThresholdExample
    "be X° on date?"exact (==)X"be 20°C on June 1?" → exact, 20
    "be X° or higher?"gte (≥)X"be 29°C or higher?" → gte, 29
    "be X° or below?"lte (≤)X"be 22°C or below?" → lte, 22
    "be between X-Y°?"range (∈)[X, Y]"be between 68-69°F?" → range, [68, 69]
    Unparseable——unclear — can't parse market rules

    3b. Bracket & tie rules

    The res_data field in ancillary_data defines which outcome wins at each threshold boundary: p1: 0, p2: 1, p3: 0.5. This means:

    • For exact: if value == threshold → p2 (Yes). If value ≠ threshold → p1 (No).
    • For gte: if value ≥ threshold → p2. If value < threshold → p1. At the boundary, p2 wins (res_data p2: 1).
    • For lte: if value ≤ threshold → p2. If value > threshold → p1.
    • For range: if value ∈ [X, Y] → p2. If value outside → p1.

    The p3: 0.5 signals a 50/50 tie. This is only used when the market is genuinely ambiguous — not as a default for uncertainty (that's unclear).

    3c. Comparison

    Apply the operator to the normalized value. The comparison is pure math on known numbers.

    OperatorNormalized value = 18, Threshold = 20Verdict
    exact18 == 20 → falseNo → p1
    gte18 ≥ 20 → falseNo → p1
    lte18 ≤ 20 → trueYes → p2
    range18 ∈ [68, 69] → falseNo → p1

    If the comparison result is unambiguous → verdict is Yes or No. If the value equals a threshold boundary and bracket rules don't clearly assign it → p3: 50/50. If the operator can't be applied (e.g., range comparison on a monthly precip value that's a single number, not a range) → unclear.

    Output — ReconciliationVerdict

    FieldDescription
    verdictYes | No | p3_tie | p4_too_early | unclear
    gateWhich gate produced the verdict: finality, quality, comparison
    normalized_valueThe value that was compared (carried forward from normalization)
    operatorThe parsed operator: exact, gte, lte, range
    thresholdThe parsed threshold value(s)
    comparison_resulttrue | false | null (if gated before comparison)
    confidence_penaltiesList of soft quality flags that reduce confidence, each with penalty amount
    reasoningConcise explanation: "RJTT recorded a low of 18°C on June 1. The market asked whether the low would be exactly 20°C. 18 ≠ 20 → No."
    Why rule parsing is regex, not LLM

    The question title follows constrained patterns driven by Polymarket's question templates. "Will the [measurement] in [city] be [threshold] on [date]?" is structurally consistent across markets. Regex patterns cover all known variants. If a title doesn't match any known pattern, it gates to unclear rather than invoking an LLM — the cost of an unparseable market rule is low (one unclear case), and the risk of an LLM hallucinating an operator is higher.

    The RetrievalSpec's unit and measurement fields provide cross-validation: if the regex extracts a threshold of "20" from a title that says "20°C", and the spec says unit=F, the mismatch is flagged and the case gates to unclear.

    res_data bracket rules — where they come from

    The res_data string is embedded in ancillary_data: res_data: p1: 0, p2: 1, p3: 0.5. Where p1 corresponds to No, p2 to Yes, p3 to unknown.

    This defines the payout values for each outcome, which doubles as the tie-breaking rule: p2: 1 means Yes wins at the boundary. p1: 0 means No only wins when the comparison is definitively false. p3: 0.5 is reserved for genuine 50/50 ties (rare — most markets say "MUST only resolve to p1 or p2").

    These are parsed with a simple regex: p1: (\d+\.?\d*), p2: (\d+\.?\d*), p3: (\d+\.?\d*).

    Decision

    src/decision/resolver.py, confidence.py, reviewer.py

    What it does

    Takes the ReconciliationVerdict and produces the final recommendation (p1–p4 or unclear) with a confidence score. The core mapping is deterministic — a lookup table from verdict to recommendation. An LLM reviewer is invoked only as a safety net when confidence is borderline.

    Step 1: Deterministic mapping

    The reconciliation verdict maps directly to a recommendation. This handles the vast majority of cases — weather thresholds are math, not judgment.

    Reconciliation verdictRecommendationWhen this happens
    Nop1Comparison returned false. The observed value does not meet the threshold.
    Yesp2Comparison returned true. The observed value meets the threshold.
    p3_tiep3The value is exactly at a boundary that bracket rules can't resolve. Rare.
    p4_too_earlyp4Finality gate failed. The next-day datapoint doesn't exist yet.
    unclearunclearA hard gate failed upstream: anomalous data, unparseable rules, retrieval exhausted.

    Step 2: Confidence calculation

    Confidence is computed from objective, auditable factors — not a model's subjective assessment.

    FactorValueRationale
    Base: API path0.95Direct API response with structured data.
    Base: Playwright path0.85Scraped data; slightly higher risk of extraction errors.
    Base: Replay path0.90Pre-captured fixture; trust the original but data may be stale.
    Penalty: per soft quality flag-0.08Each flag (partial_data, observation_gap, unit_provenance_conflict, near_day_boundary) reduces confidence. Multiple flags stack additively.
    Penalty: completeness < 0.75-0.10Significantly fewer observations than expected.
    Floor0.50Confidence cannot drop below 0.50 from penalties alone. Below this threshold, the case should have gated to unclear in reconciliation.

    Example: API path (0.95) + partial_data flag (-0.08) + unit_provenance_conflict (-0.08) → 0.79. This is Medium tier — still resolvable, but the LLM reviewer will be invoked.

    Step 3: LLM reviewer (conditional)

    The LLM is a reviewer, not a decider. It cannot override a high-confidence deterministic result. It can only escalate a borderline result to unclear if it finds a reasoning flaw.

    ConfidenceLLM reviewer invoked?What it can do
    ≥ 0.85No— The deterministic result stands. High-confidence API responses with no quality flags don't need review.
    0.70 – 0.85Yes — single LLMThe LLM receives the full evidence chain: RetrievalSpec, source trace, normalized observations, reconciliation reasoning, and the deterministic recommendation. It is asked: "Do you agree with this conclusion? If not, explain specifically what is wrong." If the LLM disagrees → escalate to unclear. If it agrees → the deterministic result stands, confidence unchanged.
    0.50 – 0.70Yes — single LLMSame review process. Additionally, if the LLM agrees, confidence is capped at 0.70 — the system is fundamentally uncertain, and the LLM's agreement doesn't make it more certain.
    < 0.50N/AShould have gated to unclear in reconciliation. If it reaches decision, treat as unclear.

    The LLM reviewer is a one-way safety valve: it can only make the system more cautious (deterministic result → unclear), never more confident (unclear → p1/p2) and never change the recommendation (p1 → p2). This preserves the conservative bias: when in doubt, admit uncertainty.

    Output — Resolution

    FieldDescription
    recommendationp1 | p2 | p3 | p4 | unclear
    confidence0.0 – 1.0 — computed from objective factors, not model-assigned
    pathdeterministic | deterministic + llm_reviewed | llm_escalated_to_unclear
    llm_reviewnull if not invoked. Otherwise: {model, agreed: true|false, reasoning: "…"}
    reasoningConcise explanation carried forward from reconciliation, optionally augmented by LLM review notes
    Why the LLM is a reviewer, not a decider

    The deterministic path (verdict → recommendation) is provably correct for the vast majority of markets: the comparison is math on known numbers. Introducing an LLM as a decider would add latency, cost, and non-determinism to cases that don't need it.

    The reviewer pattern gives us the best of both: high-confidence cases skip the LLM entirely (fast, cheap, deterministic). Borderline cases get a second opinion. But the LLM can only veto — it can't override a clear No with a Yes, because that would mean the LLM hallucinated a correction to correct math.

    This is also why there's no "quorum" of multiple models. A quorum implies the models are voting on the answer, which means they could collectively override the deterministic result. A single reviewer that can only escalate to unclear preserves the conservative bias without introducing voting dynamics.

    What the LLM reviewer sees

    The LLM prompt includes:

    • The market question (from question_data.title)
    • The RetrievalSpec (station, window, measurement, unit, precision)
    • The source trace summary (API/Playwright, status, obs count)
    • The normalized value and quality flags
    • The reconciliation reasoning: operator, threshold, comparison result
    • The deterministic recommendation and confidence

    The prompt asks: "Review this weather market resolution. The system has determined [recommendation] with [confidence] confidence. Do you see any errors in the evidence chain, logical flaw in the comparison, or reason this market should be unclear instead? Answer ONLY 'agree' or 'unclear' with a brief explanation."

    This constrained output format prevents the LLM from proposing alternative recommendations or introducing new reasoning paths. It's a binary check: agree or escalate.

    Output modes

    Live mode
    Writes output/results.json. Fixtures were already saved by the retrieval stage (step 16). The results file references the fixture by case_id.
    Replay mode
    Writes output/results.json. All data came from fixtures — the results file is identical to what the original live run would have produced.
    Evaluation: comparing results against gold answers

    An external evaluation script (evaluate.py) compares output/results.json against gold_visible/answers.json. Metrics:

    • Accuracy: % of cases where recommendation matches gold
    • Conservatism score: unclear cases that would have been wrong if forced — these are good outcomes, not errors
    • False confident: cases where the resolver returned p1/p2 with confidence ≥ 0.7 but was wrong — these are the worst failures
    • False unclear: cases where the resolver returned unclear on a gold case that was clearly resolvable

    The evaluation script runs separately from the resolver — it's a grading tool for evaluators, not part of the resolution pipeline.
